// Architecture Decision · Why Two Services?

Your backend is an infrastructure layer,
not a web server

The decision to separate backend from frontend is exactly right — and it's one of the most important structural decisions in this project. Your backend doesn't serve visitors. It runs AI agents for multiple businesses simultaneously, around the clock. Vapi webhooks arrive mid-call whether anyone is visiting your site or not. The ingestion worker processes uploaded PDFs continuously. Credits get deducted when calls end. None of this is triggered by a human clicking a page.

Express.js — always-on backend Next.js — frontend control panel only MongoDB Atlas — persistent data Cloudflare R2 — file storage Pinecone — vector store Vapi.ai — voice orchestration ElevenLabs — TTS Deepgram — STT Railway — backend host Vercel — frontend host
Old approach — Next.js API Routes only
Everything in one Next.js app
✗ Serverless cold-start: 200–800ms per invocation
✗ 60s max timeout — ingestion breaks silently
✗ No persistent memory — in-memory cache impossible
✗ Vapi webhook arrives while function is sleeping
✗ Background worker cannot run — no always-on process
✗ Scales with visitor traffic, not agent count
New approach — Separate Express backend
Express always-on + Next.js UI only
✓ Express is warm — zero cold start on webhooks
✓ No timeout limits — ingest PDFs for minutes
✓ In-memory caches live hours (voice list, etc.)
✓ Vapi webhook hits live Express immediately
✓ Ingestion worker runs via setInterval forever
✓ Scales with agent count, not visitor count
⚡
What "always-on" means in practice
Right now, at 3am, while zero users are on your website, your Express backend is still:
— Listening for Vapi webhooks from ongoing calls across all your users' agents
— Running the ingestion worker loop, checking for pending docs every 10 seconds
— Holding the ElevenLabs voice list in memory so agents can start calls instantly
— Ready to deduct credits and write full transcripts the moment any call ends

The frontend going offline doesn't interrupt any of this. Think of it like a factory — the factory floor (Express) runs 24/7. The office (Next.js) is only open when someone sits down to use it.

FLOW 1 User Onboarding & Agent Setup
FRONTEND-TRIGGERED

This flow only happens when a user is actively on the platform. Every step is initiated by someone clicking in the Next.js UI. The Express backend handles the actual work.

1
Register or log in
Login form → POST {API_URL}/api/auth/register or /login. Express validates credentials, hashes password with bcrypt, creates a User in MongoDB with 50 free credits, returns a JWT token. NextAuth stores this token in a secure session cookie. All future API calls include Authorization: Bearer {token} automatically via the Axios interceptor.
2
Create an Agent via the 4-step wizard
The wizard accumulates choices in Zustand draft state — role (Sales / Support / Tech / Custom), business description (becomes the system prompt), ElevenLabs voice selection, and temperature. On "Create Agent": POST /api/agents → Express creates an Agent in MongoDB → calls Vapi API to provision an assistant (registers our webhook URL, sets voice/transcriber/model config) → saves the returned vapiAgentId. The agent is now live in Vapi and ready to take calls.
Vapi receives: voice: 11labs {voiceId} · transcriber: deepgram nova-3 model: gpt-4o with system prompt from business description serverUrl: https://api.voiceforge.app/api/calls/webhook
3
Upload knowledge documents
User drags PDFs or DOCX onto the knowledge page. Next.js sends multipart/form-data to POST /api/knowledge/upload. Express receives the file via multer (in memory as a Buffer — never touches disk), uploads it to Cloudflare R2, creates a KnowledgeDoc with status: 'pending', and deducts 5 credits. Returns immediately. The frontend polls GET /api/knowledge/status/:docId every 3s to show progress.
4
Background worker ingests the document
The setInterval worker inside Express polls MongoDB every 10 seconds. Finds the pending doc, downloads it from R2 via presigned URL, runs LangChain (PDFLoader → RecursiveCharacterTextSplitter → OpenAI embeddings), upserts all chunks into Pinecone under the user's namespace, and sets status to 'ready'. The frontend polling sees the change and shows a green "Ready · 124 chunks" badge.
5
Optionally scrape web URLs
User pastes a URL (product page, competitor site, company blog). POST /api/knowledge/scrape → Express fetches with Cheerio, strips nav/footer/scripts, extracts clean text, uploads to R2. Same pipeline as file upload. 2 credits deducted. Worker ingests it identically.
6
Buy more credits (currently stub)
Billing page shows 3 packs (Starter ₹199 / Growth ₹799 / Business ₹2499). Clicking "Buy Now" calls POST /api/credits/purchase. Currently stubbed — Express adds credits without real payment verification. When Razorpay is wired in, the HMAC signature from Razorpay's webhook will be verified before any credits are added.
FLOW 2 Live Call Workflow — Most Important Flow
BACKEND-DRIVEN · NO UI REQUIRED
⚑

This entire flow runs while the user may not be on the website at all. The Express backend handles every step autonomously. This is the core reason the backend must always be running.

1
User triggers an outbound call
From the agent detail page, user enters a phone number and clicks "Start Call". Next.js sends POST /api/calls/outbound { agentId, toNumber }. Express validates, checks credits ≥ 5, finds the agent's vapiAgentId, calls Vapi's API to initiate the call, creates a CallLog with status: 'initiated', and returns the vapiCallId immediately to the frontend.
2
Vapi dials via Twilio
Vapi uses the Twilio number assigned to the assistant to place the PSTN call. We write zero Twilio code — Vapi manages the entire telephony layer. The callee's phone rings. When they answer, the call session begins and Vapi starts the real-time STT → LLM → TTS loop.
3
Callee speaks → Deepgram transcribes
The callee's voice streams in real-time to Deepgram Nova-3 (plugged into Vapi as the STT provider). Deepgram returns a text transcript in under 300ms. Vapi now needs relevant context before generating a response, so it fires our context webhook.
4
Vapi fires our context webhook — backend must respond fast
Vapi sends POST /api/calls/webhook with event type function-call, passing the transcribed text as the query and the userId in parameters. This must respond in under 500ms or Vapi skips context. Express calls retrieveContext(userId, query, 5) — embeds the query, queries Pinecone in the user's namespace, returns the top 5 most semantically relevant document chunks back to Vapi.
Pinecone query: embed query text → vector search in user namespace → top-5 chunks (score > 0.6) Latency target: ≤ 400ms round-trip · Express being warm = no cold start = achievable
5
GPT-4o generates the agent's response
Vapi combines the retrieved chunks + the full conversation history and sends it to GPT-4o. The system prompt (built from the agent's role and business description) instructs the model to stay grounded in the context and be concise. GPT-4o generates the agent's reply — typically 1–2 sentences for natural phone conversation pacing.
6
ElevenLabs speaks the response
The reply text goes to ElevenLabs TTS using the voice the user selected in the wizard. ElevenLabs synthesizes audio and streams it back through Vapi → Twilio → to the callee's phone. End-to-end latency (callee finishes speaking → hears AI response) is typically 600–900ms, which sounds natural on a phone call.
7
Call ends → backend handles all cleanup
When either party hangs up, Vapi sends an end-of-call-report webhook to Express. The handler: calculates durationSec, computes creditsUsed = ceil(durationSec/60) × 3, updates the CallLog with the full transcript and status: 'completed', atomically deducts credits from the user, and writes a CreditLedger entry. The frontend polling the call status eventually sees completed.
CallLog updated: status='completed' · transcript[] · durationSec · creditsUsed · endedAt User credits: $inc({ credits: -creditsUsed }) — atomic MongoDB update CreditLedger: { type:'deduct', amount: -N, description: 'Outbound call Xs to +91...' }
FLOW 3 RAG Ingestion Workflow
ASYNC · WORKER-DRIVEN

Document ingestion is intentionally asynchronous. The API returns immediately after uploading to R2. The actual chunking, embedding, and Pinecone indexing happens in the background worker — which can take 10 seconds to several minutes for large PDFs. The frontend shows a live status badge by polling.

1
File arrives at Express via multer
Multipart form data → multer puts the file Buffer in memory (never written to disk — safer and works on any host). Validated: must be pdf/docx/txt, under 10MB. 5 credits are checked and deducted immediately — this prevents uploads without sufficient balance before any expensive work begins.
2
Buffer uploaded to Cloudflare R2
Uploaded to key users/{userId}/docs/{docId}-{filename} using the AWS SDK (R2 is S3-compatible). The worker later downloads this same file from R2 — on AWS S3 that egress would cost money. On R2 it's free, which is why we chose Cloudflare R2.
3
KnowledgeDoc created as 'pending'
MongoDB record: { userId, r2Key, filename, type, status: 'pending' }. API returns { docId, status: 'pending' } to the frontend immediately. User sees a spinner. The actual ingestion hasn't started yet.
4
Worker picks it up within 10 seconds
The setInterval(tick, 10000) loop running inside Express finds the pending doc. Sets status to 'processing'. Downloads from R2 via presigned URL. Feeds it to the right LangChain loader: PDFLoader for PDFs, DocxLoader for Word files, plain string for TXT/scraped content.
5
Chunking and embedding
RecursiveCharacterTextSplitter splits into 1000-character chunks with 150-character overlap. Overlap prevents context from being lost at chunk boundaries. OpenAI text-embedding-3-small converts each chunk to a 1536-dimension vector. The same embedding model must be used at query time — mixing models breaks semantic similarity completely.
6
Vectors upserted to Pinecone with user namespace
All chunk vectors go to Pinecone under namespace = userId.toString(). This namespace isolation means Company A's documents are completely invisible when Company B's agent queries Pinecone — critical for a multi-tenant platform. Pinecone creates the namespace automatically on first upsert.
Namespace = userId → complete data isolation per customer user_abc → namespace "abc" | user_xyz → namespace "xyz" These namespaces never cross-contaminate during similarity search
7
Doc marked 'ready' — agent can use it immediately
KnowledgeDoc updated: { status: 'ready', chunkCount: N, pineconeNS: userId }. Frontend polling sees the change and shows a green "Ready · 124 chunks" badge. The next call this agent makes will now retrieve from these chunks.
FLOW 4 Credits Lifecycle
LEDGER SYSTEM

Credits are the platform's internal currency. Every debit and credit is recorded in the CreditLedger collection for audit purposes. The balance on the User document is for fast reads; the ledger is the source of truth for history.

ℹ

Why call credits are deducted at the end, not the start: We don't know the call duration until it ends. A pre-flight check (≥5 credits) prevents zero-credit calls from starting, but the real deduction matches the actual duration. The end-of-call webhook fires from Vapi and is handled entirely by Express — no UI involved, works even if the user has closed the browser.

// Raw body MUST be registered BEFORE express.json()

WORKER Ingestion Worker — Why setInterval, Not a Loop
NON-BLOCKING DESIGN 
// src/worker/index.ts
// Called once from index.ts: startWorker()
// Runs inside the same Express process — no extra deployment

export function startWorker(intervalMs = 10_000) {
  const tick = async () => {
    const pending = await KnowledgeDoc.find({ status: 'pending' }).limit(3)
    for (const doc of pending) {
      await ingestDocument(doc._id.toString())
    }
  }
  tick() // run once immediately on startup
  return setInterval(tick, intervalMs) // then every 10s
}

// WHY setInterval and not while(true) with await sleep():
// setInterval is non-blocking. Node.js continues processing HTTP requests,
// webhook calls, and database queries between ticks.
// A while(true) loop would block the event loop during await, freezing Express.
VAPI How Vapi Connects to Our Backend
WEBHOOK-BASED INTEGRATION

Vapi is the voice orchestration layer. We give Vapi three things: a TTS voice (ElevenLabs), an STT provider (Deepgram), and a serverUrl pointing to our Express backend. During calls, Vapi calls our serverUrl for real-time context and to report end-of-call events.

📤
When we call Vapi
Agent creation (POST /assistant), outbound call trigger (POST /call/phone), agent config update (PATCH /assistant/:id), agent delete (DELETE /assistant/:id).
📥
When Vapi calls us
function-call (get RAG context mid-call), status-update (call state changed), end-of-call-report (full transcript + duration). All arrive at POST /api/calls/webhook.
⏱
Latency requirement
The function-call webhook must respond in under 500ms. Express being warm (no cold start) makes this achievable. Pinecone query takes ~80–150ms typically.

FE What Next.js Actually Does
voiceforge-web/ — CONTROL PANEL ONLY

Next.js in this architecture is a React app with SSR and a clean URL structure. It does not touch MongoDB. It does not process files. It does not call Vapi. It calls one place for everything: the Express backend at NEXT_PUBLIC_API_URL. There is exactly one Next.js API route — /api/auth/[...nextauth] — which exists solely to handle the NextAuth session cookie machinery.

🖥
Server Components
Dashboard overview, call detail, agent detail — rendered server-side using a serverFetch() helper that reads the backend JWT from the NextAuth session and attaches it to the fetch call.
⚡
Client Components
Agent wizard, voice picker, call initiator with live timer, file uploader with polling, credit badge. All use Zustand for local state and SWR for data fetching.
🔒
NextAuth — JWT Bridge Only
NextAuth exists only to store the Express-issued JWT in a secure HttpOnly cookie. It calls /api/auth/login on Express, gets back a token, and gives the browser a session. That's its entire job.
AUTH The JWT Auth Bridge — Exactly How It Works 
// Step 1: User logs in on Next.js
signIn('credentials', { email, password })
  → NextAuth calls Express: POST http://localhost:4000/api/auth/login
  → Express validates password, signs JWT, returns { token, user }
  → NextAuth stores { backendToken: token } in its session JWT
  → Browser receives a secure HttpOnly session cookie

// Step 2: Browser makes any API call (lib/api.ts Axios interceptor)
const session = await getSession()
config.headers.Authorization = `Bearer ${session.backendToken}`
  → Express receives the token, verifyToken() checks the signature
  → req.user = { userId, email } is attached to the request
  → Route handler proceeds with verified identity

// Step 3: Server-side page rendering (for fast initial load)
const session = await auth()  // NextAuth server helper
const data = await fetch(`${process.env.API_URL}/api/agents`, {
  headers: { Authorization: `Bearer ${session.backendToken}` }
})

PHASES Build Phases

Phase 1 — MVP (Build Now)

Backend:   Express setup → config → DB models → JWT auth routes
           → Agents CRUD + Vapi provisioning → R2 + knowledge routes
           → Voice call routes + webhook handler → RAG pipeline
           → Credits service → Ingestion worker → Voices proxy

Frontend:  Next.js scaffold → Axios API client + JWT bridge
           → Auth pages → Dashboard shell + 21st.dev landing hero
           → Agent wizard (4 steps) → Knowledge uploader + polling
           → Call initiator + status machine → Billing page (stub payment)

Deploy:    Backend → Railway (always-on) · Frontend → Vercel
           Local webhook testing: ngrok http 4000

Phase 2 — Growth

- Real Razorpay webhook verification (replace stub in credits.routes.ts)
- Inbound call handling (Vapi phone number assignment per agent)
- Firecrawl API (replace Cheerio for JS-rendered SPAs)
- Scheduled outbound campaigns (cron via node-cron inside Express)
- Post-call sentiment analysis (GPT-4o-mini on transcript after call ends)
- Team accounts (multiple users per workspace, shared knowledge namespace)

Phase 3 — Scale

- BullMQ + Redis (replace setInterval worker for higher ingestion throughput)
- Custom voice cloning (ElevenLabs Voice Lab API)
- Real-time call monitoring (WebSocket stream of live transcript to frontend)
- CRM integrations (HubSpot, Salesforce via Zapier or direct API)
- Analytics dashboard with conversion tracking per agent
- White-label / custom subdomain support
